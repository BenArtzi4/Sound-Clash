# Observability — I-Vitals (T1.7)

The Grafana layer on top of the Faro telemetry that went live on 2026-07-12
(PR #258). Three output channels, all built and validated against **real
production data** from `https://www.soundclash.org`:

1. **Display** — the `Sound Clash - Vitals` Grafana dashboard.
2. **Push (email)** — a native Grafana Cloud email alert for the systemic #254
   signal, to `benartzi4@gmail.com`.
3. **Push (GitHub)** — a scheduled Action that files one GitHub issue per #254
   incident.

Everything here is applied **by the maintainer**: the Grafana MCP in this repo
is read-only (`-disable-write`, Viewer token), so this directory ships
importable JSON/YAML + click-by-click steps rather than writing to Grafana
directly.

## Stack & datasources

- Grafana Cloud: `https://prudentcurrant2518.grafana.net`
- `grafanacloud-logs` (Loki) — Faro logs/events/measurements
- `grafanacloud-traces` (Tempo) — Faro custom spans
- `grafanacloud-prom` (Prometheus) — currently only `grafanacloud_feo11y_app_info`
  (see [supabase-metrics-scrape.md](supabase-metrics-scrape.md))

### Faro data shape (what the queries rely on)

Faro tags every signal `service_name="sound-clash-web"`,
`deployment_environment="production"`. Stream labels are only `kind`
(`event` / `log` / `measurement` / `exception`), `detected_level`,
`service_name`. **Loki lines are logfmt, not JSON** — everything else needs
`| logfmt`. App context keys are prefixed `context_`, Faro event data
`event_data_`.

| Signal | kind | Key fields (after `| logfmt`) |
|---|---|---|
| `stale_buzz_lock_resynced` (warn) | log | `context_game_code`, `context_stale_team`, `context_fresh_team`, `session_id`, `page_id` |
| `realtime_disconnect` (warn) | log | `context_game_code`, `context_sub_status` |
| `realtime_fanout` | event | `event_data_event_type` (game/team/round), `event_data_fanout_ms` |
| web-vitals | measurement | `type="web-vitals"`, `lcp` / `inp` / `fcp` / `ttfb`, `context_rating` |
| Tempo spans | — | `game.buzz.e2e` (`outcome`, `realtime.fanout_ms`), `game.score.e2e`, `game.song_start` (`preloaded`), `db.rpc` (`rpc.name`), `game.player_init` |

## 1. Dashboard — `Sound Clash - Vitals`

- File: [`dashboards/sound-clash-vitals.json`](dashboards/sound-clash-vitals.json)
  (generated by [`dashboards/generate_vitals_dashboard.py`](dashboards/generate_vitals_dashboard.py)
  — edit the generator and re-run `python dashboards/generate_vitals_dashboard.py`).
- 21 panels: buzz/score e2e p50/p95/p99, DB-RPC latency by function, song-start
  latency **by `preloaded`** + stage breakdown, Realtime fan-out lag, the
  **#254 dropped-event repairs** panels (stale repairs + disconnects by
  `game_code`), Core Web Vitals p75 (LCP/INP/FCP/TTFB), active sessions,
  exceptions, and warning/error tails.

**Import:** Grafana → Dashboards → **New → Import** → upload
`sound-clash-vitals.json` → the datasources resolve by UID
(`grafanacloud-logs` / `grafanacloud-traces`) → Import. It lands with uid
`sound-clash-vitals`.

> The `game.score.e2e` panel has no data until a real game scores — there was no
> prod scoring traffic in the 2026-07-12 validation window. Its query is
> identical in shape to the validated `game.buzz.e2e` panel.

## 2. Email alert — systemic #254

- File: [`alerts/alerting-provisioning.yaml`](alerts/alerting-provisioning.yaml)
- Rule `sc_stale_buzz_lock_254`: fires when **any one `game_code` self-heals ≥3
  dropped Realtime buzz-lock events in 30 min**, evaluated every 5 min. A lone
  repair is one flaky phone; ≥3 in one game is an incident. Contact point
  `email-benartzi` → `benartzi4@gmail.com` (native Grafana Cloud email).

**Apply — pick one:**

- **UI (contact point):** Alerting → **Contact points → Add contact point** →
  name `email-benartzi`, integration **Email**, address `benartzi4@gmail.com` →
  Test (you get an email) → Save.
- **UI (rule):** Alerting → **Alert rules → New / Import** → paste the `groups:`
  block, or build by hand: datasource `grafanacloud-logs`, query
  `sum by (context_game_code) (count_over_time({service_name="sound-clash-web", kind="log"} |= "stale_buzz_lock_resynced" | logfmt [30m]))`,
  reduce **last**, threshold **IS ABOVE 2** (= ≥3), evaluate every **5m**,
  folder **Sound Clash**, `for: 0`.
- **UI (route):** Alerting → **Notification policies → New nested policy** →
  matcher `team = sound-clash` → contact point `email-benartzi`, group by
  `alertname, context_game_code`.
- **Or via the provisioning API** with an Editor SA token — see the header of
  `alerting-provisioning.yaml`.

**Validated:** the rule query evaluated at 2026-07-12 20:00 UTC returns
`3VX6QJ=3, AJJFJD=2, VHG4S4=2`, so it fires for exactly `3VX6QJ` (the real #259
dropped-frame harness incident) and not the two 2-repair games.

The second requested alert — **Realtime connections ≥150/200 + message quota** —
is **blocked**: Supabase metrics are not scraped into `grafanacloud-prom`.
The scrape job + alert design is in
[`supabase-metrics-scrape.md`](supabase-metrics-scrape.md).

## 3. GitHub Action — durable #254 incident record

- File: [`../.github/workflows/stale-buzz-lock-scan.yml`](../.github/workflows/stale-buzz-lock-scan.yml)
- Every 15 min it runs the same #254 query against Loki (via the Grafana
  datasource proxy, read-only token), and files **one issue per incident**
  (`game_code` with ≥3 repairs in 30 min) — with occurrence timestamps,
  stale→fresh team pairs, affected `session_id`s, and a Grafana deep-link.
  Deduped by exact title (`Realtime dropped-event incident: game <CODE> (<date>)`,
  label `stale-buzz-lock`) so reruns and the overlapping cadence never
  double-file.

### Secret to set

Mint a **read-only (Viewer) service-account token** in Grafana
(Administration → Service accounts → add, role **Viewer** → add token) and set it
as a repo secret:

```
GRAFANA_READ_TOKEN   =  <the Viewer service-account token>
```

(`gh secret set GRAFANA_READ_TOKEN --repo BenArtzi4/Sound-Clash`)

The token only ever queries the Loki datasource through the proxy — no write
scope. Validated: the exact proxy path
`/api/datasources/proxy/uid/grafanacloud-logs/loki/api/v1/query` returns the
same `3VX6QJ=3, AJJFJD=2, VHG4S4=2` result a Viewer token can read.

### Test before trusting the schedule

Actions → **Stale buzz-lock scan → Run workflow**:
`dry_run = true`, `end_time = 2026-07-12T20:00:00Z`. It will report it *would
file* exactly the `3VX6QJ` incident (3 repairs) and print the full issue body,
without creating anything. Then a real (`dry_run = false`) run against a live
incident, or just let the schedule take over.
